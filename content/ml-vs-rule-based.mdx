Working on the Sentiment Analysis Web App forced me to make hard engineering choices: should this be a machine learning model or a set of if-then rules? Here's what I concluded.

## The Problem

The app needed to classify text sentiment (positive/negative) in real time with high accuracy and low latency. Single-line inputs and multi-line text blocks. Responsive frontend. REST API backend.

Accuracy target: &gt;60%.

## Why ML Won This Round

For this problem, machine learning was the right choice:

1. **Subjectivity & Context**: "This sucks" could be negative OR sarcastic appreciation. Pure keyword rules fail on context.
2. **Variability**: Sentiment markers constantly evolve. What is slang today is normal tomorrow.
3. **Coverage**: I would need thousands of hand-written rules to match what a trained model learns automatically.

## The Solution: TF-IDF + SVM

I chose **TF-IDF (Term Frequency-Inverse Document Frequency) + SVM (Support Vector Machine)** for these reasons:

**TF-IDF**: Converts raw text into numerical vectors, emphasizing rare, discriminative words.
- Fast to compute
- Interpretable (we can see which words matter)
- Works well with limited data

**SVM**: A classical ML classifier that finds the optimal boundary between positive and negative examples.
- Highly effective for text classification
- Relatively simple training pipeline
- Predictable inference time

## The Architecture

```
1. Text Input → Tokenization
2. TF-IDF Vectorization (sparse vectors)
3. SVM Classifier (trained on labeled examples)
4. REST API (FastAPI) → JSON response
5. Frontend (JavaScript) → Display sentiment
```

## Performance

- **Accuracy**: ~65% on test set
- **Inference time**: &lt;50ms per request
- **Deployment**: Single FastAPI server, minimal dependencies

## The Hybrid Approach in Practice

What made this production-ready:

- **ML Core**: SVM for the actual sentiment prediction
- **Rules Layer**: Validation logic (empty input handling, multi-line aggregation)
- **Preprocessing**: Lowercase, punctuation removal, stopword filtering

The rules handled edge cases; the ML model handled the hard work.

## Future Improvements

The current system works well but has limits:
- No neutral class (only positive/negative)
- No live data integration
- No visual dashboards

Next iteration: Add BERT-based embeddings for better context understanding, integrate real-time data streams, and build interactive dashboards.

## The Practical Lesson

Don't default to ML. Start with the simplest solution. When patterns are too complex for rules—when context matters, when the input space is high-dimensional, when the rules would need constant updating—that is when ML earns its place.

Good engineering is matching the solution to the problem, not chasing sophistication.

But when you do need ML, choose algorithms that are fast, interpretable, and proven. TF-IDF + SVM is not flashy, but it works.
